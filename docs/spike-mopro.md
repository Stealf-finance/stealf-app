# Spike тАФ Mopro FFI build chain (Phase 0)

CEO directive: validate that the ZK build chain works on `stealf-app`
before committing to slices that depend on it (Slice 5 тАФ Stealth/Umbra).

## Background

`front-stealf` ships `modules/mopro-ffi/` (~82 MB, prebuilt
`MoproFfiFramework.xcframework`). That folder is the **build source**
for the npm package `@umbra-privacy/rn-zk-prover` тАФ it is *not* what
runs in production. Production resolves the pod via:

```
# front-stealf/ios/Podfile.lock
MoproFfi:
  :path: "../node_modules/@umbra-privacy/rn-zk-prover"
```

So replicating the runtime setup on `stealf-app` only requires the
public npm package тАФ the 82 MB module folder stays in `front-stealf`.

A live proof run is deferred to Slice 5 тАФ pulling Umbra `.zkey`
assets just for a Phase-0 smoke would be wasted effort. The Turbo
Module loads on boot (autolinking + native lib registration via
`installRustCrate()`), which is the part that could realistically
break across a repo migration. That risk is now closed.

## Verdict

ЁЯЯв **Ship Phase 0.** The ZK build chain is reproducible on `stealf-app`
with the same wiring as `front-stealf`. Slice 5 can begin from a
known-good foundation.

## Accepted risk тАФ deferred to Slice 5

**Phase 0 stopped at autolink + boot. First live `prove()` call deferred
to Slice 5.** The original CEO directive asked for a "minimal end-to-end
proof run". What was validated kills the most likely failure mode (native
build / linker / Turbo Module registration broken after repo migration),
not the ZK runtime itself.

**Risk accepted by CEO at Phase 0 close**: ZK runtime (`.zkey` loading,
witness generation, snark prove) is **not validated** on `stealf-app`
yet. If Slice 5 hits ZK-runtime weirdness, the cost of the late discovery
is on this deferral decision тАФ pivot trigger is config-level (zkey path,
asset bundling, runtime params), not architectural. Anchored here for
memory.
