# Spike — Mopro FFI build chain (Phase 0)

CEO directive: validate that the ZK build chain works on `stealf-app`
before committing to slices that depend on it (Slice 5 — Stealth/Umbra).

## Background

`front-stealf` ships `modules/mopro-ffi/` (~82 MB, prebuilt
`MoproFfiFramework.xcframework`). That folder is the **build source**
for the npm package `@umbra-privacy/rn-zk-prover` — it is *not* what
runs in production. Production resolves the pod via:

```
# front-stealf/ios/Podfile.lock
MoproFfi:
  :path: "../node_modules/@umbra-privacy/rn-zk-prover"
```

So replicating the runtime setup on `stealf-app` only requires the
public npm package — the 82 MB module folder stays in `front-stealf`.

A live proof run is deferred to Slice 5 — pulling Umbra `.zkey`
assets just for a Phase-0 smoke would be wasted effort. The Turbo
Module loads on boot (autolinking + native lib registration via
`installRustCrate()`), which is the part that could realistically
break across a repo migration. That risk is now closed.

## Verdict

🟢 **Ship Phase 0.** The ZK build chain is reproducible on `stealf-app`
with the same wiring as `front-stealf`. Slice 5 can begin from a
known-good foundation.
